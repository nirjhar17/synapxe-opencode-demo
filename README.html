Step 3 Detail: Model Serving Setup

3a. Patch KServe to Headed Mode

Why Headed mode?
Headed mode enables Gateway API integration with llm-d routing. Headless mode (default) only supports basic KNative serving without intelligent routing. 
oc patch dsc default-dsc --type=merge \
  -p '{"spec":{"components":{"kserve":{"rawDeploymentServiceConfig":"Headed"}}}}'

3b. Create Namespace and Prerequisites

# Namespace
oc create namespace ai-serving

# PVC for model cache (persists across pod restarts)
oc apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: model-cache
  namespace: ai-serving
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 100Gi
  storageClassName: gp3-csi
EOF

# HuggingFace token for model download
oc create secret generic hf-token \
  --namespace=ai-serving \
  --from-literal=token=YOUR_HF_TOKEN

3c. TLS Certificate and Gateway

# Generate self-signed cert
openssl req -x509 -nodes -days 3650 -newkey rsa:2048 \
  -keyout /tmp/tls.key -out /tmp/tls.crt \
  -subj "/CN=llm-d-gateway" \
  -addext "subjectAltName=DNS:llm-d-gateway-data-science-gateway-class.ai-serving.svc.cluster.local"

# Create TLS secret
oc create secret tls llm-d-gateway-tls \
  --namespace=ai-serving \
  --cert=/tmp/tls.crt --key=/tmp/tls.key

# Create Gateway
oc apply -f - <<'EOF'
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: llm-d-gateway
  namespace: ai-serving
spec:
  gatewayClassName: data-science-gateway-class
  listeners:
  - name: https
    port: 443
    protocol: HTTPS
    tls:
      mode: Terminate
      certificateRefs:
      - name: llm-d-gateway-tls
EOF

3d. LLMInferenceService

API Version: v1alpha2
The cluster runs RHOAI 3.3 which uses serving.kserve.io/v1alpha2. Manu's repo uses v1alpha1 — fields were renamed: modelSpec→model, routerSpec→router, workloadSpec.template.spec→template. Always verify with oc explain. 
oc apply -f llminferenceservice.yaml

See llminferenceservice.yaml in this directory for the full manifest.

Verification Commands

# Check LLMInferenceService status
oc get llminferenceservice -n ai-serving

# Check pods
oc get pods -n ai-serving

# Check all prerequisites
oc get ns ai-serving
oc get pvc -n ai-serving
oc get secret hf-token llm-d-gateway-tls -n ai-serving
oc get gateway -n ai-serving

Issues Encountered and Resolutions

Issue 1: v1alpha1 vs v1alpha2 Schema Mismatch

Symptom: invalid model URI: (empty URI)

Root cause: Manu's YAML uses v1alpha1 fields (modelSpec, routerSpec) which don't map to v1alpha2 (model, router).

Fix: Rewrote YAML to v1alpha2 schema using oc explain llminferenceservice.spec.

Issue 2: EPP EndpointPickerConfig Not Recognized

Symptom: Scheduler pod CrashLoopBackOff — no kind "EndpointPickerConfig" is registered for version "epp.llm-d.ai/v1alpha1"

Root cause: The EPP binary on RHOAI 3.3 uses a different config API version than Manu's repo.

Fix: Removed custom scheduler.config section. Controller uses default EPP config. With 1 replica, EPP scoring is not needed.

Issue 3: TLS Certificate CN Too Long

Symptom: ASN1_mbstring_ncopy:string too long:maxsize=64

Root cause: Full DNS hostname exceeded 64-char CN limit.

Fix: Used short CN (/CN=llm-d-gateway) with full hostname in SAN extension.

Issue 4: TLS Job Permission Error

Symptom: Failed to create: /var/cache/yum/metadata in UBI container.

Fix: Generated TLS cert locally with openssl and created secret directly via oc create secret tls.

Key Learnings

  • Always check CRD API version with oc explain <resource>.spec before applying YAMLs from external repos
  • RHOAI GUI cannot set nodeSelector, tolerations, PVC mounts, probe timeouts, or EPP scorer weights
  • v1alpha2 significantly flattened the LLMInferenceService schema
  • Permission denied warnings during HF model download are cosmetic (non-root container, chmod fails but write succeeds)
  • EPP config API version differs between llm-d releases — remove custom config for defaults when version mismatches
  • Model cached on PVC loads in seconds on subsequent deployments vs minutes for initial download

Request Flow

Current (1 replica, no EPP): Client ──▶ llm-d Gateway (Envoy) ──▶ HTTPRoute ──▶ vLLM Pod (Qwen3-Coder-30B) Scaled (N replicas, with EPP): Client ──▶ llm-d Gateway (Envoy) ──▶ EPP Scheduler ──┬──▶ vLLM Pod 1 (picks best ├──▶ vLLM Pod 2 replica) └──▶ vLLM Pod N EPP Scoring: - Prefix cache affinity (weight 3) ── reuse cached prompt prefixes - KV cache utilization (weight 2) ── pick replica with most free memory - Queue depth (weight 2) ── pick replica with shortest queue
