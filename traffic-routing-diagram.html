Issue & Fix Details

The Issue

The llm-d Gateway (Envoy proxy) was unable to forward requests to the vLLM model server, causing OpenCode to hang with no response.

Root Cause: HTTP/2 Protocol Mismatch

  • The Gateway terminates incoming TLS on port 443
  • A DestinationRule tells Envoy to re-encrypt traffic to the backend using TLS SIMPLE mode
  • During the upstream TLS handshake, Envoy negotiates ALPN and selects HTTP/2 (h2)
  • vLLM runs on uvicorn, which only supports HTTP/1.1 over TLS
  • vLLM rejects the HTTP/2 framing and sends a TCP RST
  • Envoy logs: 503 DR http2.remote_reset
  • OpenCode receives no response and the model appears dead

Gateway Logs (Before Fix)

# Every request to the gateway returned 503 with http2 reset [2026-07-04T10:45:25] "GET /ai-serving/qwen3-coder/v1/models HTTP/2" 503 DR http2.remote_reset - upstream: qwen3-coder-kserve-workload-svc.ai-serving.svc.cluster.local - response_flags: DR (Downstream Reset)

The Fix

Patched the DestinationRule to force HTTP/1.1 for all upstream connections to the vLLM backend service.

Command Applied

oc patch destinationrule qwen3-coder-kserve-workload-svc \ -n ai-serving --type=merge -p '{ "spec": { "trafficPolicy": { "connectionPool": { "http": { "h2UpgradePolicy": "DO_NOT_UPGRADE" } } } } }'

What This Does

  • h2UpgradePolicy: DO_NOT_UPGRADE tells Envoy to never upgrade connections to HTTP/2 when connecting to the backend
  • Envoy now connects to vLLM using HTTP/1.1 over TLS
  • vLLM (uvicorn) accepts HTTP/1.1 perfectly
  • The existing TLS SIMPLE config with service-ca.crt still handles encryption

Verification

# After the fix, the gateway returns 200 with model data $ curl -sk https://llm-d-gateway-.../ai-serving/qwen3-coder/v1/models {"object":"list","data":[{"id":"Qwen/Qwen3-Coder-30B-A3B-Instruct-FP8",...}]} $ curl -sk -X POST .../v1/chat/completions \ -d '{"model":"Qwen/Qwen3-Coder-30B-A3B-Instruct-FP8","messages":[...]}' {"choices":[{"message":{"reasoning":"Hello!"}}]} # Model responds!

Issue 2: Model Responds But OpenCode Shows Nothing

The Issue

After fixing the gateway routing, the model was reachable and responding. However, OpenCode displayed a blank response after "Thinking" — the user typed "hi" but got no visible output, just repeated "Thought" and "Compaction" cycles.

Root Cause: Reasoning Parser Splitting Output

  • vLLM was launched with --reasoning-parser qwen3 in the VLLM_ADDITIONAL_ARGS
  • This flag tells vLLM to split Qwen3's output into two fields: reasoning (thinking tokens) and content (final answer)
  • Qwen3-Coder is a reasoning model — it puts its entire response inside the thinking block by default
  • Result: "content": null and "reasoning": "Hi there! How can I help you?"
  • OpenCode only reads the content field — it sees null and displays nothing

API Response (Before Fix)

{ "choices": [{ "message": { "role": "assistant", "content": null, ← OpenCode reads this (empty!) "reasoning": "Hi there! How can I help you?" ← Actual response is here } }] }

Additional Issue: Token Limit

  • OpenCode defaults to requesting max_tokens: 32000 for custom providers
  • Model's max_model_len is 32768
  • Even a small prompt (769 input tokens) causes: "you requested 32000 output tokens and your prompt contains at least 769 input tokens, for a total of at least 32769 tokens"
  • OpenCode kept retrying in an infinite "Compaction" loop

The Fixes

Fix A: Remove Reasoning Parser from vLLM

Removed --reasoning-parser qwen3 from the LLMInferenceService so vLLM puts all output in the content field.

oc patch llminferenceservice qwen3-coder -n ai-serving \ --type=json -p '[{ "op": "replace", "path": "/spec/template/containers/0/env/0/value", "value": "--disable-uvicorn-access-log --max-model-len 32768 --gpu-memory-utilization 0.90 --enable-prefix-caching --enable-auto-tool-choice --tool-call-parser qwen3_coder --kv-cache-dtype fp8" }]' # Removed: --reasoning-parser qwen3 # Kept: --tool-call-parser qwen3_coder (tool calling still works)

Fix B: Set Token Limits in OpenCode Config

Added limit block to the model definition in opencode.json to cap output tokens at 8192.

// ~/.config/opencode/opencode.json "models": { "Qwen/Qwen3-Coder-30B-A3B-Instruct-FP8": { "name": "Qwen3 Coder 30B (Private AI)", "limit": { "context": 32768, "output": 8192 } } }

API Response (After Fix)

{ "choices": [{ "message": { "role": "assistant", "content": "Hi there! How can I help you today?", ← Response is here now! "reasoning": null } }] }

Note on Reasoning Models

  • Qwen3-Coder supports a /no_think system prompt to disable thinking without removing the parser
  • However, removing the parser entirely is cleaner for OpenCode since it doesn't understand the reasoning field
  • Tool calling still works via --tool-call-parser qwen3_coder (separate from reasoning)
  • If reasoning/thinking is needed later, re-add --reasoning-parser qwen3 and use a client that supports the reasoning field
